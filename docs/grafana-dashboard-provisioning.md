# How the Grafana dashboard got set up (sidecar provisioning)

Worked example: [`grafana-log-viewer`](../grafana-log-viewer)'s **App Logs (sample-nginx)**
dashboard. Same mechanism applies to any Grafana install using the `grafana/grafana` chart
(this one, via its `loki-stack` dependency) — not specific to logging or Loki.

## The mechanism

Grafana itself has no idea any of this is happening — it just polls a local folder on disk
for `.json` dashboard files, same as if you'd hand-copied them in. A second container in the
pod (a "sidecar") is what actually gets files into that folder, by watching the Kubernetes API
rather than the filesystem:

```
you: kubectl apply a ConfigMap, labeled grafana_dashboard: "1"
        │
        ▼
grafana-sc-dashboard container (quay.io/kiwigrid/k8s-sidecar)
  watches the K8s API (not a filesystem) for ConfigMaps/Secrets with that label,
  in this namespace
        │  on match: writes the ConfigMap's data (dashboard JSON) to a shared volume
        ▼
emptyDir volume, mounted at /tmp/dashboards in BOTH containers
        │
        ▼
grafana container's own file-based provisioner (sc-dashboardproviders.yaml,
  auto-generated by the chart) polls /tmp/dashboards every 30s
        │
        ▼
dashboard appears under Dashboards — no Grafana restart, no manual import,
  no UI click-through
```

Confirmed on the live pod (3 containers once this is enabled):

```
$ kubectl get pod -n log-viewer <grafana-pod> -o jsonpath='{range .spec.containers[*]}{.name}{"\n"}{end}'
grafana-sc-dashboard      # the sidecar — watches K8s API, writes matched dashboards to disk
grafana-sc-datasources    # same pattern, separately, for datasources (already used for Loki)
grafana                   # Grafana itself — only ever reads a local folder

$ kubectl get pod -n log-viewer <grafana-pod> -o jsonpath='{range .spec.containers[?(@.name=="grafana-sc-dashboard")].env[*]}{.name}={.value}{"\n"}{end}'
METHOD=WATCH
LABEL=grafana_dashboard
LABEL_VALUE=1
FOLDER=/tmp/dashboards
RESOURCE=both

$ kubectl exec -n log-viewer <grafana-pod> -c grafana -- cat /etc/grafana/provisioning/dashboards/sc-dashboardproviders.yaml
apiVersion: 1
providers:
- name: 'sidecarProvider'
  type: file
  options:
    path: /tmp/dashboards
```

## What was actually added

**1. Turn the sidecar on**, in `grafana-log-viewer/values.yaml` — `label`/`labelValue` here
must match the ConfigMap's label exactly, or the sidecar's `WATCH` never matches it:

```yaml
grafana:
  sidecar:
    dashboards:
      enabled: true
      label: grafana_dashboard
      labelValue: "1"
```

**2. Ship a dashboard as a `ConfigMap`**, in `grafana-log-viewer/templates/dashboard-nginx-logs.yaml`
— a normal chart template, rendered and applied by Helm like any other resource:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: {{ .Release.Name }}-dashboard-nginx-logs
  labels:
    grafana_dashboard: "1"          # <- must match values.yaml's label/labelValue above
data:
  nginx-logs.json: |
    { ... dashboard JSON model ... }
```

Nothing Loki-specific about the ConfigMap mechanism — any dashboard JSON works. What's specific
to *this* dashboard is inside the JSON:

```json
{
  "panels": [{
    "type": "logs",
    "datasource": { "type": "loki", "uid": "$datasource" },
    "targets": [{ "expr": "{app=\"nginx\"}", "datasource": { "type": "loki", "uid": "$datasource" } }]
  }],
  "templating": {
    "list": [{ "name": "datasource", "type": "datasource", "query": "loki" }]
  }
}
```

The `$datasource` template variable (type `datasource`, query `loki`) is the portable-dashboard
trick: it resolves to *whichever* Loki datasource Grafana has, by plugin type, at render time —
instead of a hardcoded datasource UID that would only happen to work because
`grafana.sidecar.datasources` provisioned Loki with that exact UID on this exact install. A
dashboard exported from Grafana's UI ("Export → for sharing externally") produces this same
shape for exactly this reason.

## Verifying it, without opening a browser

The same principle as the rest of this saga (see
[`incidents.md`](./incidents.md#2026-07-21-grafana-log-viewer-stuck-pending-insufficient-cpu)):
check the data layer directly rather than trusting that a UI *should* show something.
Grafana's own HTTP API confirms whether the sidecar → provisioner → Grafana chain actually
worked, no browser needed:

```bash
kubectl exec -n log-viewer deploy/log-viewer-grafana -c grafana -- \
  wget -qO- 'http://admin:admin@localhost:3000/api/search?query='
# [{"uid":"app-logs-nginx","title":"App Logs (sample-nginx)","url":"/d/app-logs-nginx/...", ...}]
```

Present in that list = provisioned successfully. Absent after 30s+ = check the sidecar
container's own logs (`kubectl logs <pod> -c grafana-sc-dashboard`) for a label mismatch or a
JSON parse error, before assuming Grafana itself is broken.

## Adding another dashboard

Copy the pattern, not the specifics — one new template file, no `values.yaml` change needed
(the sidecar is already watching):

```bash
cp grafana-log-viewer/templates/dashboard-nginx-logs.yaml \
   grafana-log-viewer/templates/dashboard-<name>.yaml
```

Then: change the `ConfigMap` `name`, keep the `grafana_dashboard: "1"` label, give the
dashboard JSON a different `uid` (must be unique — reusing one silently overwrites the other
dashboard), and change the panel's `expr` to whatever Loki query targets your app (`{app="..."}`,
`{namespace="...", container="..."}`, etc. — check available labels with
`kubectl exec -n log-viewer log-viewer-loki-0 -- wget -qO- http://localhost:3100/loki/api/v1/label`,
same as used to verify ingestion in [`incidents.md`](./incidents.md)). `helm upgrade` applies
the new `ConfigMap`; the sidecar picks it up within ~30s on its own.
